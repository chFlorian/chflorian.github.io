Introduction to SwiftData

The Swift-first solution for on-device persistance in SwiftUI.

For over a decade, CoreData was the go-to first-party solution for on-device persistence. The framework also supports CloudKit, allowing us developers to "easily" sync data between the user's iCloud-enabled devices. Custom store URLs allow sharing the database with an app's extensions (like widgets).

SwiftData is the SwiftUI of Persistence. CoreData is like UIKit. Still relevant, but lots of apps can benefit from the simpler APIs and better integrations that the newer framework offers. Let's dive in!

Creating models

What used to be done in confusing model files, is now implemented right in Swift. Let me give you an example:

import SwiftData

@Model
class Book {
    var name: String
    var releaseDate: Date 
}

This is a very simple example, you're right. But it actually doesn't get much more complex. Let's briefly cover two important concepts: Attributes and Relationships.

Attributes allow you to change the behavior of properties (i.e. making them unique or getting them indexed in Spotlight). Have a look at this expanded example that makes sure the name property is always unique:

import SwiftData

@Model
class Book {
    @Attribute(.unique) var name: String
    var releaseDate: Date 
}

To store a Relationship between books and authors, let's have a look at this example:

import SwiftData

@Model
class Book {
    @Attribute(.unique) var name: String
    var releaseDate: Date 
 
    var author: Author?
}

@Model
class Author {
    var name: String

    var books: [Book]
}

No, you don't need to explicitly add the @Relationship macro to your relationships in most cases. To learn more about this behavior, I recommend reading the documentation and this article by Paul Hudson.

App-level setup

Now that we have successfully created our models, it's time to tell our app struct about them.

import SwiftData
import SwiftUI

@main
struct MyBookApp: App {
    var body: some Scene {
        WindowGroup {
            ContentView()
        }
        .modelContainer(for: [
            Book.self,
            Author.self
        ])
    }
}

Technically, you don't need to specify all types if they appear in a relationship with another specified type. I prefer to be explicit here and always list all @Model types.

Reading data

Reading data is one of my favorite parts about SwiftData and it's quite similar to CoreData:

import SwiftData

struct ContentView: View {
    @Query(sort: \.releaseDate, order: .reverse) var books: [Book]

    ...
}

It's a tight integration into SwiftUI's state architecture and re-evaluates the View.body whenever an entry changes, gets added or gets deleted.

Creating data

To create new entries, we need a ModelContext. Luckily, the .modelContainer SceneModifier we added to our WindowScene inserts this context in the Environment. We can easily grab it like this:

@Environment(\.modelContext) private var context

Lastly, we just need to create a model instance and insert it into the context:

let book = Book(name: "Atomic Habits", releaseDate: releaseData, author: jamesClearAuthor)
context.insert(book)

Saving is handled implicitly by the framework, but you can always manually call context.save() as well.

That's it! SwiftData is easy to get started. It handles CloudKit automatically if you follow the steps in the documentation. App Groups are also supported out-of-the-box by simply enabling the App Group capability in your target settings.
