Stop Using Padding Everywhere

Padding is my most-used ViewModifiers. In these 3 situations I prefer better alternatives.

Don't get me wrong, it's totally fine to use .padding(). I use it a lot as well! And in the first few versions of SwiftUI, there really weren't many alternatives. But now there are, so let's explore them.

Stacks

What I used to do: 

VStack {
  FirstView()
    .padding(.bottom, 16)

  SecondView()
}

For two Views in one Stack, this might seem fine. Imagine this setup for multiple Views - all of them getting modified to have padding at the bottom. The semantics are important: .padding() modifies the View to have empty space at the bottom. This is not the same as creating Space between Views. To create space between the Views in any Stack, you should use the spacing: parameter:

VStack(spacing: 24) {
  FirstView()

  SecondView()
}

Please note that the default spacing of Stacks (if you ommit the spacing parameter entirely) changes depending on context. In my tests on an iPhone 17 simulator running iOS 26.0, it's round about 8. That's why the second code sample uses 24 points of spacing (16 + 8).

ScrollViews

We all know this annoying problem with ScrollViews: We apply a .padding() to the ScrollView, so everything inside is moved away from the device edges. Unfortunately, this also moves the scrollbar away from the edge, making it cover the content. Of course you could just apply the padding to the scroll content, but there's a more contextual alternative:

// "Wrong" Example

ScrollView {
  ...
}
.padding()

// "Correct" Example
ScrollView {
  ...
}
.contentMargins(16, for: .scrollContent)

Using content margins makes it more explicit, which is almost always a good thing (regarding readability of your code). You can use the same modifier to move the scroll bar as well, just use .scrollIndicators instead.

Lists

This one is similar to ScrollViews in that it adds context, but it also serves a functional purpose. Consider this "incorrect" example of using padding to inset content in a list row:

List {
  FirstView()
    .padding(.leading, 8)
}

Once again, this might read fine at first. I already hinted at the contextual issue here: You don't really want to make the View larger (= add padding). Instead, you want to inset the View from its parent (the list row, a View structure implicitly created by the List). So the more contextual API is this:

List {
  FirstView()
    .listRowInsets(.leading, 8)
}

Important: The default value is actually not 0 - it's something like 8. This means that using a value below 8 allows you to move the View closer to the device edge than normally. This is the added functionality I hinted at before.
